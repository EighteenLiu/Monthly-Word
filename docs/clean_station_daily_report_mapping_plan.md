# 清洁站日报生成逻辑修改开发文档

## 1. 需求理解

本次修改只针对“密闭式清洁站”日报生成逻辑，不影响中转站日报和月报逻辑。

现有清洁站日报模板在“2. 具体情况”中循环输出 `station.items`，每个 item 的标题来自台账的 `2级指标` 或 `具体问题`。这会导致日报正文标题直接显示指标名，例如“标志不完整不清晰、喷涂不规范”“收集车辆敞口运输”等。

目标是：清洁站日报正文不要直接写这些指标名，而是把台账中的问题指标映射到以下 6 个固定展示标题，并把这些标题写进 Word 日报：

- 正门及门牌
- 公告及文件
- 运输车辆
- 内部环境
- 灭火器
- 操作规范及流程

“整体情况”部分仍然需要总结当前站点存在的问题，不能只看 `具体问题` 文本是否为空，要结合 JSON 中“密闭式清洁站”下的 2 级指标和 3 级指标逐一判断。

## 2. JSON 指标结构

参考文件：`E:/实习/standard_indicators_extracted.json`

其中“密闭式清洁站”部分包含两类内容。

第一类是用于判断问题的 2/3 级指标：

| 2级指标 | 3级指标 | 判断含义 |
| --- | --- | --- |
| 标志不完整不清晰、喷涂不规范 | 无问题 / 有问题 | `有问题` 为问题，`无问题` 为非问题 |
| 收集车辆敞口运输 | 无问题 / 有问题 | `有问题` 为问题，`无问题` 为非问题 |
| 未开门运行 | 无问题 / 有问题 | `有问题` 为问题，`无问题` 为非问题 |
| 无称重系统或称重系统损坏 | 无问题 / 有问题 | `有问题` 为问题，`无问题` 为非问题 |
| 小型收集车混装混运 | 无问题 / 有问题 | `有问题` 为问题，`无问题` 为非问题 |
| 箱体内垃圾混投 | 无问题 / 有问题 | `有问题` 为问题，`无问题` 为非问题 |
| 拒收单不准确 | 无问题 / 有问题 | `有问题` 为问题，`无问题` 为非问题 |
| 良好，未发现问题 | 无 3 级指标 | 整体非问题标记，不应进入具体问题标题 |

第二类是日报正文展示标题：

| 展示标题 |
| --- |
| 正门及门牌 |
| 公告及文件 |
| 运输车辆 |
| 内部环境 |
| 灭火器 |
| 操作规范及流程 |

我的理解是：JSON 里后 6 项不是问题判断项，而是清洁站日报模板中“具体情况”的固定栏目名称。代码需要把第一类问题指标归入这 6 个栏目。

## 3. 建议映射关系

根据指标语义，建议先采用以下映射：

| 台账 2级指标 | 映射到日报标题 | 理由 |
| --- | --- | --- |
| 标志不完整不清晰、喷涂不规范 | 正门及门牌 | 对应站点外部门牌、标识喷涂、标志规范性 |
| 收集车辆敞口运输 | 运输车辆 | 对应车辆运输方式 |
| 小型收集车混装混运 | 运输车辆 | 对应收集车辆混装混运问题 |
| 箱体内垃圾混投 | 内部环境 | 对应站内箱体、投放秩序和内部环境 |
| 拒收单不准确 | 操作规范及流程 | 对应拒收单填写和流程执行 |
| 未开门运行 | 操作规范及流程 | 对应运行管理和操作流程 |
| 无称重系统或称重系统损坏 | 内部环境 | 用户确认归入内部环境 |

目前 JSON 中没有直接可映射到“公告及文件”“灭火器”的问题判断指标。建议这两个栏目仍作为固定展示标题保留，但只有当后续台账出现可匹配指标或补充规则后才填充照片/问题。

已确认口径：

- “无称重系统或称重系统损坏”归入“内部环境”。
- “未开门运行”归入“操作规范及流程”。
- “公告及文件”“灭火器”没有遗漏的其他指标来源，但要支持台账指标本身就是这 6 个固定标题的情况。

## 4. 问题判断规则

清洁站问题判断应改为结合 `2级指标` 和 `3级指标`：

1. 先只处理 `2级点位` 归一化后等于 `密闭式清洁站` 的台账记录。
2. 若 `2级指标` 是 JSON 中列出的清洁站问题判断指标，并且 `3级指标` 为 `有问题`，则认为该记录是问题记录。
3. 若 `3级指标` 为 `无问题`，则不认为是问题，即使 `2级指标` 名称本身看起来像问题描述。
4. `2级指标 = 良好，未发现问题` 永远作为非问题记录。
5. 若 `3级指标` 为空，且 `2级指标` 不在上述问题判断指标中，默认不进入问题总结，避免误写固定展示标题。
6. 如果台账 `具体问题` 有更细描述，则“整体情况”优先使用 `具体问题`；否则使用 `2级指标` 文案作为问题描述。
7. 同一站点重复问题需要去重，保持当前摘要格式：`存在的问题是：（1）...；（2）...。`

这意味着当前 `normalizer.build_summary([record.issue_text ...])` 不够精确，需要为清洁站新增基于完整 `LedgerRecord` 的摘要构建逻辑。

## 5. 具体情况生成规则

清洁站“2. 具体情况”不再按原始 `station.items` 逐条输出指标名，而是输出映射后的固定栏目。

建议新增清洁站专用的模板数据，例如：

- `station.detail_items`
- 每个 detail item 包含：
  - `name`: 固定标题，如 `正门及门牌`
  - `items`: 归入该标题的原始问题 item 列表
  - `photos`: 合并后的照片
  - `photo_rows`: 两张一行的照片排版数据
  - `issue_texts`: 该标题下的问题文本

为了避免日报出现空标题，有两种实现策略：

方案 A：只输出有问题或有照片的栏目。

优点：日报更简洁。缺点：六个固定标题不一定全部出现。

方案 B：始终输出 6 个栏目，没有问题和照片的栏目只显示标题。

优点：结构稳定。缺点：空栏目可能显得多余。

采用方案 A：只输出有问题或有照片的栏目。

## 6. 模板是否需要修改

需要修改清洁站日报模板，但改动不大。

当前模板核心结构是：

```jinja2
{%p for station in clean_stations %}
{{ station.index_cn }}、 {{ station.level3_point }}{{ station.level4_point }}密闭式清洁站
1. 整体情况
{{ station.summary }}
2. 具体情况
{%p for item in station.items %}
（{{ loop.index }}）{{ item.name }}
{%p if item.photo_rows %}
{%p for row in item.photo_rows %}
{{ row.left }}{% if row.right %}{{ row.right }}{% endif %}
{%p endfor %}
{%p endif %}
{%p endfor %}
{%p endfor %}
```

建议按方案 A 修改模板：具体情况循环从 `station.items` 改为 `station.detail_items`。

```jinja2
{%p for station in clean_stations %}
{{ station.index_cn }}、 {{ station.level3_point }}{{ station.level4_point }}密闭式清洁站
1. 整体情况
{{ station.summary }}
2. 具体情况
{%p for item in station.detail_items %}
（{{ loop.index }}）{{ item.name }}
{%p if item.photo_rows %}
{%p for row in item.photo_rows %}
{{ row.left }}{% if row.right %}{{ row.right }}{% endif %}
{%p endfor %}
{%p endif %}
{%p endfor %}
{%p endfor %}
```

“具体情况”不显示每个标题下的问题文字，只显示标题和照片。因此模板不需要 `issue_texts/issue_summary`。

```jinja2
{%p for item in station.detail_items %}
（{{ loop.index }}）{{ item.name }}
{%p if item.photo_rows %}
...
{%p endif %}
{%p endfor %}
```

中转站模板不用改，因为中转站已经有自己的 `detail_items` 逻辑。

## 7. 代码实现方案

建议新增一个清洁站映射模块：

`daily_report_generator/services/clean_station_mapping.py`

内容包括：

- 固定标题列表：`正门及门牌`、`公告及文件`、`运输车辆`、`内部环境`、`灭火器`、`操作规范及流程`
- 问题指标集合：JSON 中 7 个带 `有问题/无问题` 的 2 级指标
- 2级指标到展示标题的映射表
- `is_clean_station_problem(record)`：结合 `indicator_name` 和 `indicator_result` 判断是否为问题
- `clean_station_problem_text(record)`：优先返回 `issue_text`，否则返回 `indicator_name`

修改 `aggregator.py`：

- 当前 `aggregate_type()` 对所有点位统一使用 `build_summary([record.issue_text ...])`
- 改为：
  - 中转站：保持原逻辑
  - 清洁站：调用新的 `build_clean_station_summary(station_records)`
- 清洁站 `build_items()` 可以保留原始 item 数据，但 item 的问题性后续由渲染层二次判断。

修改 `renderer.py`：

- `station_to_template()` 中：
  - 中转站：继续使用 `build_transfer_station_detail_items(items)`
  - 清洁站：改用 `build_clean_station_detail_items(items)`
- 清洁站 detail item 按映射标题聚合照片和问题文本。
- 每个标题下照片去重，继续沿用两张一行的 `photo_rows` 结构。

修改 `models.py` 的建议：

- 当前 `StationItem` 没有保存原始 `2级指标/3级指标/具体问题` 的完整判断上下文，只保存了部分字段。
- 已有字段基本够用：`indicator_name`、`result`、`issue_text`、`photo_paths`。
- 如要支持一个展示标题下多个原始问题文本，建议 detail item 使用 `SimpleNamespace` 即可，不必改 dataclass。

## 8. 验收标准

1. 清洁站日报“具体情况”中的标题只能出现以下固定标题之一：
   - 正门及门牌
   - 公告及文件
   - 运输车辆
   - 内部环境
   - 灭火器
   - 操作规范及流程
   - 良好，未发现问题
2. 不再出现原始指标名作为标题，例如：
   - 标志不完整不清晰、喷涂不规范
   - 收集车辆敞口运输
   - 小型收集车混装混运
   - 箱体内垃圾混投
   - 拒收单不准确
3. `3级指标 = 无问题` 的记录不进入“整体情况”问题总结。
4. `3级指标 = 有问题` 的记录进入“整体情况”问题总结。
5. `良好，未发现问题` 不进入问题总结，也不进入具体情况标题。
6. 照片仍能按映射后的标题显示。
7. 中转站日报生成结果不变。

## 9. 测试建议

建议增加纯 Python 单元测试，不依赖 Word 模板：

- 构造清洁站 `LedgerRecord`：
  - `标志不完整不清晰、喷涂不规范 + 有问题` -> summary 有问题，detail title 为 `正门及门牌`
  - `标志不完整不清晰、喷涂不规范 + 无问题` -> summary 无问题，不生成 detail item
  - `收集车辆敞口运输 + 有问题` -> detail title 为 `运输车辆`
  - `良好，未发现问题` -> 不生成问题
- 构造含照片的 `StationItem`，验证照片进入映射后的 `photo_rows`。
- 构造中转站数据，验证仍使用原中转站 detail mapping。

## 10. 已确认口径

1. `无称重系统或称重系统损坏` 映射到 `内部环境`。
2. `未开门运行` 映射到 `操作规范及流程`。
3. `公告及文件` 和 `灭火器` 当前没有其他问题指标来源，但支持台账 `2级指标` 本身为这 6 个固定标题时直接匹配。
4. 清洁站“具体情况”只显示标题和照片，不显示标题下的问题文字。
5. 采用方案 A：只输出有问题或有照片的栏目。
